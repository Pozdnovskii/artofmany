# Оценка миграции artofmany.com → Astro + Sanity

## Контекст

Запрошена оценка (не реализация) переноса сайта https://artofmany.com/ — портфолио барселонского креативного агентства (Джофре Санфелиу, офисы в Барселоне и Дубае) — на стек Astro + Sanity, на основе трёх уже готовых референс-проектов команды: **albatros-realestate**, **videostudio** и текущего **yourmove**. Цель — понять техническую сложность, переиспользуемые паттерны и риски, прежде чем оценивать трудозатраты и планировать реализацию.

Ниже — не план кода, а архитектурная оценка + scoping. Собрана через:
- разбор трёх референс-репозиториев (Explore-агенты по каждому);
- живую проверку artofmany.com (`curl` — WebFetch словил 503 от WAF сайта, curl прошёл нормально) — sitemap, HTML структура проектной страницы, CSS-классы грида.

## 1. Что представляет собой artofmany.com сейчас

- **Платформа: Cargo.site** (proprietary CMS+хостинг для портфолио-сайтов дизайнеров) — подтверждено доменами `freight.cargo.site`/`static.cargo.site`, JS-бандлом Cargo, атрибутами `data-ce-host`.
- **81 URL в sitemap.xml**: 1 главная + 80 страниц проектов (`/Optimism`, `/The-Golden-Man` и т.д.). `/about` и `/team` отдают 200, но **не входят в sitemap** — стоит уточнить у клиента, намеренно это или нет.
- **4 шаблона страниц**: Главная (автоплей-видео на фон + JS-грид превью проектов), Проект (заголовок, короткий блок credits — агентство/директор/арт-директор/клиент, галерея изображений, несколько встроенных Vimeo-iframe), About (лонгрид-био + соцссылки + 2 адреса офисов), Team (участники: имя/роль/био/контакты).
- **Фильтры категорий** (Public/Private/Brand Direction/Campaigns/Exhibitions/Editorial/Films/Lab/Archive) — это теги на одном и том же гриде, а не отдельные шаблоны/сущности.
- **Нет форм, поиска, e-commerce, аккаунтов, бронирования** — чисто контентная витрина. Контакт — только mailto/телефон.
- **Нет i18n** — сайт одноязычный (английский), переключателя языка/hreflang нет.
- **Источник данных для скрейпинга — полностью найден и подтверждён живыми запросами, headless-браузер не нужен**:
  - Каждая страница проекта содержит `<script type="text/json" data-set="ScaffoldingData">` — вложенное дерево страниц сайта. Реальные данные текущего проекта лежат по пути `.pages[<индекс PROJECTES>].pages[0]` и включают `content`/`content_partial_html` (HTML тела **с сохранёнными ссылками** — агентство/директор/клиент захардкожены как `<a href="...">`, это важно: `content_no_html`/`excerpt` эти ссылки теряют, для импорта нужно брать именно `content`/`content_partial_html`), плюс `images`, `backdrop`, `thumb`, `tags`, `title`. Всё это отдаётся обычным `curl`, без исполнения JS.
  - `<title>` и `<meta name="description">` тоже server-rendered с реальными credits (агентство/директор/арт-директор/клиент) в виде простого текста — годится как быстрый кросс-чек, но без ссылок.
  - Есть отдельный публичный JSON API со списком всех проектов разом: `https://artofmany.com/_api/v0/thumbnails/artofmany?page_id=4707125&all=true&limit=200` — один запрос отдаёт все 80 проектов (`id`, `url`/slug, `title`, `tags`, `excerpt`, `thumb`). Без `&limit=200` API отдаёт только 50 (пагинация через `&offset=`); `page_count:80` подтверждён в `ScaffoldingData` для сета `PROJECTES` (id `4707064`).
  - WAF: `robots.txt` явно разрешает краулинг всем кроме списка SEO-ботов, `Crawl-delay: 2`. Серия запросов подряд (в т.ч. пробные к несуществующим `_api/v0/*` эндпоинтам) прошла без блокировок — обычный `curl` с браузерным User-Agent отрабатывает стабильно.
- **Грид на главной — обычный responsive CSS-grid**, не bespoke-раскладка: `grid-template-columns: 1fr 1fr` на десктопе, одна колонка на мобильном (подтверждено визуально пользователем). Первичный вывод про "Cargo Montessori freeform grid с ручным позиционированием плиток" был ошибочным — он основан на присутствии CSS-классов `[thumbnails=montessori]`/`data-can-move`/`.resize-handle` в общей теме Cargo, которая по умолчанию содержит стили сразу для всех поддерживаемых режимов раскладки (grid/justify/montessori), а не на фактически используемом на этом сайте режиме. Переносится обычной адаптивной CSS-сеткой, без специального JS (см. п.5).
- Изображения раздаются по стабильному прямому URL (`freight.cargo.site/t/original/i/<hash>/<filename>`) — скачиваются без обхода.
- **Видео подтверждено напрямую (`curl`)**: на главной — обычный HTML5 `<video>` с прямым mp4 (`files.cargocollective.com/c261833/Artofmany22.mp4`, autoplay/muted/loop), Vimeo там нет. На странице проекта (`/The-Golden-Man`) — наоборот, реальные `<iframe src="//player.vimeo.com/video/<id>?...">`, по 3 штуки на этой странице, вперемешку с изображениями галереи.

## 2. Рекомендуемый стек

Ближайший референс — **videostudio**, а не albatros или yourmove: у artofmany нет i18n вообще, поэтому вся машинерия многоязычности (field-level у albatros через `translatedField()`, document-level у yourmove через `translation.metadata`) — чистый оверхед, который не нужен.

- **Astro 7**, static output, `build.format: "file"` — как в videostudio.
- **Sanity 6** через чистый `@sanity/client` (`useCdn:false`, `perspective:"published"`), без `@sanity/astro` — как в videostudio; не нужен live-preview для 81-страничного портфолио.
- **pnpm**. videostudio берётся как база (решено обсуждением), но структуру стоит сделать компактнее его версии — не копировать один в один. Раздельные `package.json` для сайта и `studio/` (паттерн videostudio, чтобы Studio-зависимости не тянулись в билд сайта) оставить как идею, но без лишней инфраструктуры videostudio (pnpm-workspace overrides под конкретные native-deps конфликты и т.п.) — решается по факту на этапе скаффолда, под реальный набор зависимостей этого проекта, а не переносится вслепую.
- **Деплой: Cloudflare Workers + Wrangler**, `prerenderEnvironment:"node"` (Sharp не работает в Workers), `imageService:"compile"` — как в videostudio. Технически сайту хватило бы и обычного Cloudflare Pages (чистая статика, форм нет), но Workers-паттерн даёт готовый flow "Deploy-кнопка в Studio → deploy hook" (`api/deploy.ts`), который команда уже умеет поддерживать.
- Плоская маршрутизация `src/pages/` без `[lang]/`: `index.astro`, `about.astro`, `team.astro`, `[slug].astro` (79 проектов через `getStaticPaths`).

## 3. Схема Sanity

Документы:
- **`project`** (80 кейсов) — `title`, `slug`, `categories` (массив тегов, как `event.tags` в videostudio), `description` — **простой rich text без форматирования (только параграфы + ссылки)**, не структурированный объект/массив credits. Изначальная идея "распарсить в `{role, people}`" отклонена сознательно: роли не образуют фиксированный enum (см. проверку ниже) и разбирать их на структурные поля не нужно — весь блок `project_content_text` ("intro + Agency: X. Director: Y...") целиком идёт в один rich text field с сохранением `<a href>` как link-аннотаций, как есть, без попытки выделить роли отдельными полями. `gallery` (массив изображений с alt), `videos` (массив Vimeo-объектов, см. п.4), `coverImage`, `order`/`date` для сортировки грида.
  - Прогнал реальную проверку по всем 80 проектам (curl, с уважением `Crawl-delay: 2`): вхождений вида `Label:` — 24 разных ярлыка (Client 65×, Creative team/Team 43+7+5×, Graphic Design 15×, Agency 6×, Interior Design 5×, Photographer/Photography 3+3×, Project Direction/Director 3+3×, и по 1× — Curator(s), Naming, Motion Graphics, Filmmaker, Copy/Copywritter, Art Direction/Director и т.д., плюс пара вероятных ложных срабатываний регэкспа не на реальных ролях). Подтверждает: набор ролей у агентства широкий и несогласованный (branding/interior/exhibition/motion — разные дисциплины), это не тот случай, где стоит городить схему под роли — обычный rich text с абзацами и ссылками полностью покрывает исходные данные без потерь.
- **`teamMember`** — name/role/bio/photo/contactLinks — почти 1:1 с `teamMember.ts` в videostudio.

Синглтоны: `homePage` (фоновое видео), `aboutPage` (био + 2 офиса), `contactInfo`/`settings` (соцссылки, без формы).

Переиспользовать из `videostudio/studio/schemas/shared.ts` целиком: `slugField`, `seoFields`, `imageWithAlt`, `richTextField`, `standardGroups`. Из yourmove — **ничего из i18n-хелперов** (`translatedField`, `hiddenOnNonEn`, `docLang` и т.п. не нужны), также не нужен `linkAnnotation()` — внутренних перелинковок в контенте нет.

**Теги/категории — SEO-решение открыто, не зафиксировано**. Варианты на столе (обсуждались, но окончательно не выбраны): индексируемые landing-страницы по дисциплинам (`discipline`-документ, аналог `service.ts`+`services/[slug].astro` в videostudio) для тегов с поисковым интентом (Brand Direction/Campaigns/Editorial/Films/Exhibitions) vs `noindex` для служебных меток (Public/Private/Archive/Lab/ALL) vs что-то другое — решить отдельно, до того как фиксировать схему тегов.

**SEO-мета — пока переносим с Cargo как есть**, без редизайна под ключевые слова. Наблюдение остаётся в силе (текущие `title`/`description` не заточены под keyword-стратегию — title вида "{Проект} — Artofmany.", description — сухой дамп credits), но решение по этому пункту отложено — возможно, будем пересматривать позже отдельным заходом, не блокирует первую версию миграции.

**Заголовки (h1/h2)**: у текущего сайта нет нормальной семантической структуры заголовков (Cargo рендерит текст через generic `div`/`class`, не `<h1>`/`<h2>`). Пока не обязательное требование к первой версии — просто зафиксировано в бэклоге как желательное улучшение (доступность + SEO-гигиена), можно сделать `sr-only` там, где видимый заголовок не нужен по дизайну.

**Иконки для соцссылок**: у Cargo это приватный icon-font (`font-family:Icons`, `static.cargo.site/assets/social/IconFont-Regular-0.9.3.woff2`) — общий framework-ассет Cargo для соцсетей/UI-кнопок, а не что-то специфичное для сайта. Переносить его не нужно — соцссылки в `contactInfo` описываются как `{platform: "instagram"|"facebook"|...|url}`, рендерятся стандартным SVG-набором иконок по тому же принципу, что `FEATURE_ICON_NAMES`/`Icon.astro` в yourmove (`src/lib/icons.ts`) — enum с compile-time проверкой в Studio-дропдауне.

## 4. Видео (фон главной — mp4, страницы проектов — Vimeo)

Разделение подтверждено напрямую curl-проверкой (см. п.1): фоновое видео на главной — обычный `<video>` с прямым mp4-файлом, отдельного решения не требует (стандартный `<video autoplay muted loop>`, mp4-поле в схеме `homePage`). Vimeo-iframe встречаются **только на страницах проектов** (галерея вперемешку с изображениями). videostudio уже решает MP4/WebM (нативный `<video>`) и YouTube (лёгкий iframe-facade через `@astro-community/astro-embed-youtube`). Для Vimeo (только `project.videos`) нужно **написать заново**, аналогично паттерну:
- `vimeoId(url)` в `lib/media.ts` (по образцу существующего `youTubeId()`);
- собственный `VimeoEmbed.astro` — постер + click-to-load iframe (~30 строк), без новой зависимости — так же, как остальные компоненты в референсах написаны руками, а не через UI-кит.
- Постер видео — загружается редактором в Sanity вручную (у Vimeo нет надёжного public thumbnail API, в отличие от YouTube).
- **Проверить заранее**: не привязаны ли Vimeo-embeds клиента к домену `artofmany.com` (domain-restricted embed) — иначе не заработают на стейджинге.

**Хранение самих видеофайлов** (фон главной + любые прямые mp4, не Vimeo): достаточно обычного Sanity `type:"file"` — работает на любом плане, без адаптивного стриминга, но текущий сайт и так отдаёт единственный статичный mp4 без вариантов качества. У Sanity есть нативный видео-модуль **Media Library** с адаптивным HLS-стримингом, но это **платный аддон Enterprise-плана** — для проекта такого масштаба избыточен и, скорее всего, недоступен по тарифу. Внешний хостинг (Cloudflare R2) не обязателен — можно держать всё, кроме Vimeo-ссылок и hero-картинок, в Sanity как `file`-ассеты, упрощая стек на один сервис.

## 5. Грид на главной — обычная задача внутри шаблонов, не отдельный риск

Уточнено и подтверждено пользователем: это простой responsive CSS-grid (`grid-template-columns: 1fr 1fr` → 1 колонка на мобильном), без ручного позиционирования плиток и без bespoke JS. Первоначальный вывод про "hand-tuned Cargo Montessori grid" был ошибкой анализа (см. п.1) — реального подтверждения такому режиму на этом сайте не нашлось, только общие CSS-классы framework-темы Cargo.

Со стороны Sanity: `project.coverImage` + `project.order` для сортировки плиток — стандартная задача, ничем не отличается от любой другой карточной сетки в референс-проектах (например `portfolioPhotos` в videostudio). Реализуется в рамках обычного Astro/Tailwind-шаблона главной страницы (`grid grid-cols-2 max-sm:grid-cols-1`), без отдельной оценки.

## 6. Подход к переносу контента (80 проектов) — метод подтверждён, spike больше не нужен

Headless-браузер не требуется — весь пайплайн строится на плоском `curl`/HTTP-клиенте:

1. Один запрос к `_api/v0/thumbnails/artofmany?page_id=4707125&all=true&limit=200` — список всех 80 проектов (id/slug/title/tags/excerpt/thumb).
2. Для каждого слага — `curl` самой страницы проекта, парсинг `<script data-set="ScaffoldingData">`, извлечение `.pages[...].pages[0].content` (HTML тела со ссылками — credits и Vimeo-iframe уже внутри) + `images`/`backdrop`.
3. Импорт — по уже проверенному паттерну команды: скрипт на `@sanity/client` с write-токеном, `sanity.transaction()`, — аналог `scripts/migrate-blogpost-staging-to-prod.mjs` и `seed-*.mjs` в albatros-realestate. `content` (HTML) конвертируется в Portable Text (парсинг `<a>`/`<br>`/текста — сущностей немного, руками написать проще, чем тащить html-to-portable-text библиотеку целиком).
4. Картинки скачиваются с прямых `freight.cargo.site` URL и перезаливаются через `client.assets.upload()` — самый объёмный труд по времени выполнения (80 проектов × N изображений), но не по сложности — чисто скриптовая рутина.
5. About/Team — всего 2 страницы, не входят в 80 проектов (и не в `_api/v0/thumbnails`) — дешевле **внести вручную через Studio**, чем скриптовать под них отдельный парсинг.
6. Аудит структуры уже частично сделан — прогон по всем 80 `content`-блокам подтвердил широкий разброс "ролей" в описании (24 разных ярлыка, см. п.3), поэтому решили не парсить их структурно, а хранить весь блок как rich text — это снимает риск потери данных на нестандартных страницах: rich text не ломается от того, что где-то нет "Client:", а где-то есть незнакомая роль.

## 7. Грубая оценка сложности по этапам (относительный размер, не часы)

| Этап | Объём | Размер |
|---|---|---|
| Схема + Studio | project/teamMember/синглтоны, shared.ts (почти копия из videostudio), Deploy-кнопка, иконки соцссылок по паттерну yourmove | **S** |
| Шаблоны страниц Astro | Главная (простой responsive grid + mp4-фон), `[slug].astro`, About, Team, SEO/JSON-LD | **M** |
| Vimeo-компонент | `vimeoId()` + `VimeoEmbed.astro` (только для страниц проектов) | **S** |
| Импорт контента | Скрипт на найденных API/JSON-эндпоинтах (curl, без headless-браузера) + HTML→Portable Text конвертация + перезаливка картинок на 80 проектов | **S–M** |
| Деплой/CI | Wrangler + Cloudflare Workers, deploy hook, переключение DNS | **S** |

Итог: проект небольшой по меркам этих трёх референсов, без открытых архитектурных рисков — и фронтенд-часть (грид, видео), и метод извлечения контента оказались проще, чем выглядели на первый взгляд. Все пункты, требовавшие ручного spike, уже проверены live-запросами в ходе этой оценки.

## 8. Открытые риски/вопросы — прежде чем переходить к детальному плану

1. ~~Реакция WAF Cargo на скрейпинг~~ — **снято**: `robots.txt` разрешает краулинг (`Crawl-delay: 2`), серия последовательных запросов (в т.ч. к несуществующим `_api/v0/*` путям) прошла без блокировок.
2. ~~Реальный источник данных Cargo~~ — **снято**: найден и подтверждён (`ScaffoldingData` JSON-блок в HTML страницы + публичный `_api/v0/thumbnails/...` эндпоинт для списка всех 80 проектов), см. п.1 и п.6.
3. Настройки приватности/domain-restriction встроенных Vimeo-видео клиента.
4. ~~Источник фонового видео главной~~ — **снято**: файл отдаётся напрямую по стабильному URL (`files.cargocollective.com/c261833/Artofmany22.mp4`), скачивается без проблем; вопрос только в том, устроит ли клиента текущее качество или нужен мастер повыше разрешением.
5. Шрифт подтверждён — **Neue Haas Grotesk** (`NeueHaasGroteskText-{Regular,Medium,Bold,Italic,...}.woff`, отдаётся с `static.cargo.site`) — коммерческий шрифт Monotype/Linotype, лицензируется отдельно от Cargo. Файлы технически скачиваемы с CDN, но нужно уточнить у клиента, есть ли действующая лицензия на self-hosting шрифта на новом домене (у Cargo может быть отдельное сайт-лицензионное соглашение, не переносимое автоматически).
6. Осознанное ли исключение `/about` и `/team` из sitemap.xml.
7. Права на переиспользование клиентских/кампейн-изображений при переносе на новую инфраструктуру.

## Проверка/дальнейшие шаги

Это оценочный документ, не код. Ручной spike по извлечению данных уже выполнен в ходе этой оценки (см. п.1, п.6) — метод скрейпинга подтверждён живыми запросами, дополнительного исследования не требуется. Осталось получить от клиента ответы по оставшимся рискам п.8 (Vimeo-приватность, права на изображения, лицензия шрифта) до начала скрейпинга/импорта, и завести новый репозиторий + Sanity-проект под это отдельно от yourmove, когда решение о старте будет принято.
